12.1 Extra Text

The HL7 v2 standard provides only a foundation (framework) to be used as a starting point and was designed to accommodate the many diverse business processes that exist in the healthcare industry. This universal design was necessary to gain broad industry support; however, attempting to accommodate such wide-ranging requirements resulted in a standard that include numerous optional elements as well as other aspects that are not completely defined, making alignment of interface implementations overly difficult. Profiling is
For these reasons, further definition of the base standard (through constraining or extending) is necessary to support a use and context. Profiles are created to provide precisely defined requirements. Profiles reduce or eliminate the optionality (or openness) of a base standard. They also are used to manage and document requirements that are necessary and sufficient for implementation.
A key recognition is that the message profile definition is an independent entity that can be reused in other implementation guides or for other purposes.
For example, the VXU V04 (Unsolicited Vaccination Record Update) message event is a generic template for communicating information about a patient’s immunization related events. The base message template is composed of mostly optional data elements. For a given use case, e.g., Send Unsolicited Immunization Update for the US Realm [4], the message template is “profiled”. That is, elements can be constrained to be required, content can be bound to a set of pre-coordinated codes, and so on. The base message event (e.g., VXU V04) that has been constrained for a particular use (e.g., submitting immunization events) is referred to as a conformance profile. An implementation guide is a collection of conformance profiles organized for a workflow (e.g., submitting, acknowledging, querying, and responding to/for immunization events). In this example, four conformance profiles exist, each with different message events; one for submitting an immunization event, for sending an acknowledgment, for querying for an immunization history, and for providing an immunization history.
A major challenge for the healthcare industry is achieving interoperability among proprietary applications provided by different vendors. For example, each hospital department may use one or more applications that need to share clinical and administrative information with other applications. Each application may require multiple communication interfaces that must be modified and maintained. This approach is a difficult way to achieve interoperability. Alternatively, interoperability can be achieved through the use of standardized interfaces, which can reduce the cost of building and managing unique interfaces for each associated application. Developers can build applications that conform to the standardized interface definition, increasing the likelihood of interoperability and reducing cost. Maintenance cost is also reduced, since there are fewer unique interfaces to maintain. One obstacle to achieving usable standardized interfaces is that the base standards are often not defined at a granular enough level to support a specific use case or local requirements; therefore, the base standard must be further defined or “constrained”.

12.1.2 Compliance (USE THIS)

For each of the conformance constructs described above, the standard defines a set of “allowable constraints” (i.e., the rules for profile compliance) that can be applied. Allowable constraints are a means of restricting requirements in the base standard (or another profile) to make those requirements more specific. As such, elements that are “Required” cannot be relaxed, e.g., changed to “Optional”.
For example, in HL7 v2.x an element that is declared as optional (O) in the base standard can be constrained (“profiled”) to required (R). The conformance constructs and allowable constraint models are defined by each standard.

2.7

Normative lengths are only specified for primitive data types. Normative Length is specified in one of two forms:

• the minimum and the maximum length separated by two dots, e.g. m..n

• the list of possible values for length separated by commas, e.g. x,y,z

When a normative length is asserted, conformant messages must have a length that lies within the boundaries specified. The boundaries are inclusive, so a length of 1..2 means the length of the item may be either 1 or 2.

Note: The minimum length is always 1 or more. If an item is optional, and there is no content present, the item is considered as not populated, rather than present with a length of 0.

Note: The deletion indicator is treated as having no length and therefore fits to all length specifications.

Conformance Length

2.7



Constraining length follows the methodology provided in this specification and can be used in any HL7 v2 version. The length constraint only applies to primitive data elements. Although, earlier versions of HL7 v2 specified lengths for composite elements (e.g., a field with a complex data type), length constraints for composite elements is not supported in message profiles.






Although various versions of the HL7 standard length may not be normative or is ambiguous to requirements, the message profile can be used to specify normative minimum and maximum length and conformance length. Such specification in derived profiles are deemed to be normative if so specified. The HL7 v2 Conformance Methodology specifies that constraints related to length are optional and this choice can be determined on a per element basis. That is, length constraints can be specified for every primitive data element or selected primitive data elements at the discretion of the specifier. Furthermore, if a length constraint is specified, then specification must follow the rules stated in this document. Length constraints include Minimum and Maximum Length, Conformance Length, and Truncation Indicators.
