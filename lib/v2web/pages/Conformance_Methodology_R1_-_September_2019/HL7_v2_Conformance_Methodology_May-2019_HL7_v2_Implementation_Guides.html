3 HL7 v2 Implementation Guides

HL7 v2 messages are not used in a vacuum, rather they are part of a larger system designed to satisfy a use case. An HL7 v2 implementation guide is a specification that describes this larger context. An implementation guide is created to organize a collection of message profiles for specifying a set of related interactions described in a use case or use cases. Implementation guides may describe broader conformance requirements such as application functionality. Such requirements may include how a set of messages are to be used to enact certain application functionality among applications (actors).
Broadly speaking, implementation guides are containers for message profiles. In short, implementation guides describe the use case(s), the applications (actors) involved, the message profile(s), the interaction among the actors; and they also can specify functional requirements. A message profile in the context of a messaging standard describes the requirements for a single interaction (e.g., Send Immunization Historyâ€”VXU V04 message). An implementation guide often describes numerous message profile definitions.

As stated in the Conformance Clause (Section X.X), the implementation guide is informative within the context of this specification but highly recommended for documenting an interoperability solution. However, certain aspects in an instance of an implementation guide can be made normative by the implementation guide author. For example, an implementation guide can contain functional requirements and declared those as normative requirements. However, the implementation need not contain functional requirements at all.
The remainder of this section provides a typical outline of an HL7 v2 implementation guide. The layout and components provide guidance.

3.1.5 Conformance Clause

An implementation guide may contain a conformance clause that defines the requirements, criteria, or conditions that must be satisfied by the implementation to claim conformance. The conformance clause identifies what must conform and how conformance can be met. A conformance claim is a declaration by an implementer of the requirements in the conformance clause that their implementation satisfies.
The conformance clause typically indicates only the high-level conformance requirements. The requirements may be stated in terms of a profile, a set of profile components (add composite profile), or identifiable parts of a specification. Profile and profile components are mechanisms that provide a structured approach to specifying requirements. The detailed requirements are contained and delineated in the other parts of the specification. That is, a conformance clause in an HL7 v2 implementation guide could indicate a number of profiles available to the implementer and describe the components that pertain to the profiles. This is the level of detail the conformance clause would contain. The detailed requirements in support of a profile or profile component are specified in the other part of the specification.
The conformance clause will indicate the normative and informative parts of the specification. Normative parts of the implementation guide contain requirements that must be implemented (and therefore, verifiable). Informative parts of an implementation guide indicate guidance that aid in implementation or provide supplementary information. No provisions in the informative part of a specification needs to be implemented.
The conformance clause also can describe options in the form of a profile and levels. Profiles can be used to define a set of requirements within the specification. The profiles may use modular components (See Chapter 7, Principles of Effective Profiling). The conformance clause may state which profile must be implemented or provide a choice (or options) of profiles that could be implemented. For instance, some profiles may be required whereas others are optional.

The conformance clause can contain levels that indicate the various degrees in which conformance requirements can be specified. For example, the lowest level could indicate the minimum set of requirements necessary. Another level could indicate additional requirements. The highest level would indicate full conformance (that is, all provisions are met).

3.4 Interaction Specification

The Interaction Specification describes the dynamic aspects of a system. In HL7 v2, an interaction specification typically will include the sequence of trigger events and resulting message flows between two or more actors (Interaction Model), and acknowledgement responsibilities. The Interaction Specification can be represented in literal or graphical form. Graphical form can include standard UML interaction diagrams such as a sequence diagram, an activity diagram, and a collaboration diagrams. An interaction specification must be defined in an implementation guide and must include an interaction model and acknowledgement responsibilities; however, this specification does not prescribe a specific method or format for doing so.
