2 Profile Levels

Profiles innately form a logical hierarchy as levels of constraints are applied to a base standard. That is, profiling is typically an iterative process that proceeds from the base standard to intermediate levels (e.g., a national level profile), to local (implementation) levels. The three profile levels are:

  • Standard Profile

  • Constrainable Profile

  • Implementable Profile

Figure X illustrates the basic profile hierarchy and the degree of constraints applied at each level. The standard-level profile (hereafter called “standard profile”) represents the base standard definitions and constraints, and establishes the framework for a specific type of event (e.g., an unsolicited vaccination record update – VXU V04 message event). Treating the standard as a profile aids in related discussions and management. At this level, the overall structure (“pattern”), including the data type definitions, are established; however, the full declaration of requirements has yet to be specified, and therefore considerable openness still exists. For example, the allowable structure of segments is defined in the base standard, but not the complete set of segments that are to be included for use will not have been determined.
Other profile levels are subsequently derived from the standard profile. The standard profile can be defined more precisely by adding constraints for a desired use (e.g., the national level of requirements for the unsolicited vaccination record update message) to create a constrainable profile. A constrainable profile is derived from either the standard profile or another constrainable profile, and it further constrains the definition attributes in accordance to the profile compliance rules stipulated by the standard. For example, a state-level constrainable profile can be derived from a national-level profile by adding more constraints for this specific state.

Figure X: General Profile Hierarchy
In a constrainable profile, analogous to the standard profile, not all element attributes are fully constrained. An implementable profile defines all elements such that all optionality and openness have been removed. All interfaces deployed in a production setting are implementable profiles whether they are documented (explicitly stated) or not documented (implicitly stated). It is highly recommended that interfaces be completely documented to the implementable profile level using the profiling mechanisms described in this specification.
An implementable profile may also be derived by further constraining another implementable profile. In this case all openness has been removed, however, additional constraints on attributes are applied; for example, the usage “required, but may be empty” can be strengthened to “required” if the refined use case needs it to be.
As described previously, constraints can be added iteratively, thereby forming a hierarchy of profiles; but a certain set of rules must be followed. A profile can only further constrain, not relax, the requirements defined in its parent profile. For example, if an element (e.g. field) is “required” in the parent profile, the element cant be profiled to “optional” in a (compliant) child profile, because “optional” would be relaxed relative to “required”.
